# ADR 0004: フロントエンドに feature-based 構成を採用する

## 背景
React アプリの規模拡大に備え、画面や共通部品だけでなく、ユースケース単位で実装を追える構成が必要だった。

## 採用理由
- 変更理由と実装配置が対応しやすい。
- feature ごとの責務が見えやすい。
- `shared` への安易な集約を抑制しやすい。
- バックエンドの module 分割と対応づけて理解しやすい。

## 却下案
### pages 中心構成
- 初期は単純だが、ユースケースが増えると責務が混ざりやすい。

### アトミックデザインを全体構造に採用
- UI 分類には有効でも、ユースケース整理には直接つながらない。

## 影響
- `features` と `shared` の境界管理が重要になる。
- page は組み立て責務に寄せ、ロジック集約点にしない。
