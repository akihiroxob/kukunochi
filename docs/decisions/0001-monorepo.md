# ADR 0001: モノレポ構成を採用する

## 背景
API、Web、契約定義を同じプロダクトとして整合的に進めたい。初期段階では、変更の追跡と共有ルールの統一を優先したい。

## 採用理由
- API と Web の変更を同時に管理しやすい。
- `packages/contracts` を単一の契約源として扱いやすい。
- 設計文書と実装仕様を同じリポジトリで維持できる。

## 却下案
### マルチリポジトリ
- リポジトリ境界は明確になるが、初期の速度より運用コストが先に増えやすい。
- 契約変更の同期が重くなる。

## 影響
- 依存ルールを明確に保つ必要がある。
- `packages -> apps` の逆依存を防ぐ運用が重要になる。
